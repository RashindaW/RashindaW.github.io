Motivation

Why agent-driven ULM?

Traditional manual ULM processes create bottlenecks and security risks at scale.

🔓

Identity Security Crisis

78% of organizations have faced identity-related security incidents, with data breaches averaging $4.88M in cost. Manual access management is a root cause.

⚙️

Operational Bottleneck

Enterprises with thousands of users, global teams, and complex access structures face severe delays from repetitive manual provisioning and deprovisioning workflows.

🤖

Agents as the Solution

LLM agents interpret natural-language instructions, extract requirements, build structured work plans, and autonomously interact with enterprise systems while ensuring policy compliance.

Architecture

Multi-agent coordinator pattern

A hierarchical multi-agent system where a supervisor orchestrates specialized sub-agents with shared context.

Fig. 1: ULMA multi-agent architecture. The Front Agent parses user requests and forwards them to the Supervisor, which orchestrates sub-agents (Identity, Teams, Policy) via shared context. The Remote Branch communicates over the A2A protocol.

Agents

Meet the agents

Each agent has a specialized role, powered by the appropriate Gemini model for its complexity.

front_agent
gemini-2.5-flash-lite

Interacts with the human user, parses natural-language ULM requests, and forwards structured tasks to the Supervisor.

supervisor_agent
gemini-2.5-pro

The orchestrator. Formulates plans, dispatches sub-agents, monitors shared context, manages approvals via long-running operations, and reports progress.

identity_agent
gemini-2.5-flash

Connects to Microsoft Entra ID via MCP server and Microsoft Graph API. Handles user creation, application access, and all AD-related operations.

policy_agent
gemini-2.5-flash

Reads organizational policy documents (PDFs), extracts relevant constraints for each request, and ensures all actions comply with security standards.

teams_agent
gemini-2.5-flash-lite

Posts progress updates and obtains human approvals. In this demo, Teams is simulated via local text files representing incoming/outgoing messages.

remote_agent
A2A Protocol

Facilitates cross-branch communication. A mock remote agent runs on localhost, simulating a branch office with its own simplified AD (JSON-based).

Stack

Technical foundation

Built entirely on Google ADK with real enterprise integrations.

Agent Framework
Google ADK + Vertex AI
All agents built with Google ADK, powered by Gemini 2.5 models via Vertex AI.
Active Directory
Microsoft Entra ID
Real Entra ID trial instance for actual user provisioning and access management.
Tool Integration
MCP Server (FastMCP)
Local MCP server via FastMCP + uvicorn, connecting to AD through Microsoft Graph API.
Local Storage
SQLite
Stores local user records (reducing AD calls) and session memory for persistent context.
Cross-Branch
A2A Protocol
Agent-to-Agent communication for remote branch operations over localhost mock server.
Observability
ADK LoggingPlugin
Built-in trace logging plus Teams agent logs for full request lifecycle visibility.
